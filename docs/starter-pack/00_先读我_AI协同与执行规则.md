# 00｜先读我：Recruitment Dashboard v3 AI 协同与执行规则

项目名称：Recruitment Dashboard v3  
中文名：智能招聘 AI 看板 / 招聘效率管理系统  
执行对象：WorkBuddy / CodeBuddy  
状态：v2 产品级重建，不是旧项目续写，不是缩水版 MVP

---

## 1. 重建原则

v2 不是推翻旧产品，而是基于旧版 Phase 0-12 的完整经验做干净重建。

旧版已经验证过：
- 招聘业务链路成立；
- AI 辅助分析有价值；
- 周报、Action、Offer 风险、面试质量模块有业务意义；
- UI 必须走轻量 SaaS 风格；
- 临时 workspace、长上下文、大 Phase 容易导致丢失和混乱。

v2 的目标：
- 完整继承旧版所有核心模块；
- 用更清晰的架构重新实现；
- 每一步小范围推进；
- 每一步审查通过再继续；
- 所有代码持久化到 Git 仓库。

---

## 2. 系统定位

本系统是智能招聘 AI 看板，用于：

1. 招聘效率分析；
2. 岗位卡点诊断；
3. 候选人全链路管理；
4. 岗位画像校准；
5. 业务筛选反馈；
6. 面试质量分析；
7. Offer 风险提醒；
8. 入职前承诺台账；
9. 协同 Action；
10. AI 辅助判断；
11. 周报复盘；
12. 人才池沉淀；
13. 面试官赋能。

---

## 3. 系统不做什么

必须长期遵守：

1. 不替代 Moka；
2. 不替代 ATS 主流程；
3. 不发 Offer；
4. 不做 Offer 审批；
5. 不做薪酬审批；
6. 不做合同；
7. 不做入职办理；
8. 不自动录用；
9. 不自动淘汰；
10. 不自动推进候选人阶段；
11. 不自动关闭岗位；
12. 不做 AI 自动决策；
13. 不强绑定真实 Moka API；
14. 不提交真实候选人隐私；
15. 不提交真实 API Key。

---

## 4. 正式 Phase 规划

v2 不是 8 个阶段的缩水版，正式按 Phase 0 - Phase 13 推进：

- Phase 0：工程底座与持久化机制
- Phase 1：权限、角色与系统边界
- Phase 2：核心数据模型与 Seed
- Phase 3：岗位管理与 JD / 画像体系
- Phase 4：候选人与投递链路
- Phase 5：业务筛选反馈与岗位画像校准
- Phase 6：面试反馈与结构化面评
- Phase 7：协同 Action 任务中心
- Phase 8：数据导入与历史数据接入
- Phase 9：Offer 风险与入职前承接
- Phase 10：AI 辅助分析系统
- Phase 11：招聘复盘与周报
- Phase 12：Design System、UI Polish 与开源化收口
- Phase 13：面试官赋能中心

每个 Phase 单独开发、单独 commit、单独 build、单独自检、单独审查。

---

## 5. 每个 Phase 的硬性流程

```text
收到当前 Phase 指令
  ↓
只执行当前 Phase
  ↓
pnpm typecheck
pnpm lint
pnpm build
  ↓
git status
git commit
git tag
  ↓
输出自检报告 + commit hash + 截图
  ↓
等待外部审查
  ↓
审查通过后才进入下一 Phase
```

禁止：
- 一次性做多个 Phase；
- 自行提前进入下一 Phase；
- 因为“顺手”修改 API、Schema、权限；
- 只把代码留在临时 workspace；
- 没有 commit hash 就宣称完成；
- 没有 build 结果就宣称完成。

---

## 6. 每个 Phase 的自检报告必须包含

1. 当前 Phase 名称；
2. 修改文件清单；
3. 新增文件清单；
4. 是否修改 API；
5. 是否修改 Schema；
6. 是否修改权限逻辑；
7. 是否存在 mock 数据；
8. 是否存在真实敏感数据；
9. 是否新增 hex；
10. 是否替代 Moka；
11. `pnpm typecheck` 结果；
12. `pnpm lint` 结果；
13. `pnpm build` 结果；
14. `git status`；
15. commit hash；
16. 截图清单；
17. 已知问题；
18. 是否建议进入下一 Phase。

最终是否进入下一 Phase，由用户 / 外部审查决定。

---

## 7. 防丢失机制

必须执行：

```bash
git status
git add .
git commit -m "phase-x: ..."
git tag phase-x-complete
```

每个 Phase 都必须留下：
- commit hash；
- tag；
- 自检报告；
- 截图；
- 文档更新。

项目不得只存在临时 workspace。
