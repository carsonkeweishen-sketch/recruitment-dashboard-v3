## 1. 全局规则

### 1.1 设计参考优先级

所有页面设计必须按以下优先级对齐参考对象：

| 优先级 | 参考对象 | 学什么 | 不学什么 |
|--------|---------|--------|---------|
| P0 | **Panda Design System (Setproduct)** | Dashboard 布局模板、KPI 卡片三要素、组件分类体系 | 不复制其品牌色/logo |
| P0 | **Linear** | 倒 L 形布局、表面层级、减少视觉噪音 | 不学暗色默认模式（Phase 12 前） |
| P0 | **Stripe** | Design Token 驱动、间距 8px 基准、组件预设 | 不学 Stack 非标准 CSS |
| P1 | **Greenhouse** | 招聘业务 UI 模式、评分表单、Job Dashboard 左侧导航 | 不学其旧版 UI |
| P1 | **Ashby** | 一体化招聘 Dashboard、Analytics 独立模块 | — |
| P1 | **Tremor** | KPI 卡片、Data Bars、Spark Charts | 不引入组件库，只学模式 |
| P2 | **Rippling** | HR 模块分离、员工状态颜色编码 | 不学黄色品牌色 |
| P2 | **Attio** | 单像素边框、4px 网格、信息密度 | 不学极高密度暗色模式 |
| P2 | **Airtable** | 白色画布+深色墨水、pill CTA | — |
| P2 | **Vercel/Geist** | 组件三态、P3 色域 | — |
| P2 | **Untitled UI** | 完整组件库变体设计 | — |

### 1.2 通用布局结构

```
┌──────────┬──────────────────────────────────────┐
│          │  Topbar (56px)                        │
│          │  搜索框 │ 通知 │ 用户头像(姓名缩写)      │
│ Sidebar  ├──────────────────────────────────────┤
│ (240px)  │  Page Header                          │
│          │  标题 + 描述 + 主操作按钮(右侧)          │
│          ├──────────────────────────────────────┤
│          │  KPI Cards Row (4 cards max)          │
│          ├──────────────────────────────────────┤
│          │  Filter Bar                           │
│          │  分段控制器(状态) + 搜索 + 高级筛选(折叠)  │
│          ├──────────────────────────────────────┤
│          │  Main Content                         │
│          │  ┌─────────────────┬────────────────┐ │
│          │  │  DataTable      │  DetailDrawer  │ │
│          │  │  (主列表区)      │  (点击行弹出)   │ │
│          │  │                 │                │ │
│          │  └─────────────────┴────────────────┘ │
└──────────┴──────────────────────────────────────┘
```

### 1.3 页面四态强制

| 状态 | 触发条件 | 展示内容 |
|------|---------|---------|
| **Loading** | 数据请求中 | Skeleton 骨架屏（不是 spinner） |
| **Empty** | 数据为空 | 空状态插图 + 文案 + 引导操作 |
| **Error** | 请求失败 | 错误图标 + 错误信息 + 重试按钮 |
| **PermissionDenied** | 无权限 | 锁图标 + "你无权访问此页面" + 返回 |

### 1.4 数据展示三级制

| 层级 | 视觉权重 | 适用数据 | 示例 |
|------|---------|---------|------|
| 一级 | 深色加粗/品牌色 | 核心标识、关键数字 | 候选人姓名、岗位名称、KPI 数值 |
| 二级 | 灰色正文 | 辅助信息 | 部门、职级、更新时间 |
| 三级 | 浅灰/Tooltip | 元数据 | 候选人编号、数据来源、创建时间 |

## 2. Dashboard 工作台

### 2.1 页面身份

| 属性 | 值 |
|------|-----|
| **路由** | `/` |
| **服务角色** | 全部 6 个角色（按角色显示不同内容） |
| **3 秒认知** | "我今天该处理什么？招聘整体怎么样？" |
| **Phase** | Phase 3+ 持续迭代 |

### 2.2 核心 KPI

| KPI | 说明 | 角色可见性 |
|-----|------|-----------|
| 待处理事项 | 需要当前用户处理的 Action 数 | 全部 |
| 在招岗位 | 活跃招聘中的岗位数 | 全部 |
| 本周面试 | 本周安排的面试数 | admin/leader/hrbp/recruiter |
| 面试反馈逾期 | 已面试但未提交反馈的数量 | admin/leader/hrbp/recruiter |
| Offer 风险 | 当前有风险的 Offer 数 | admin/leader/hrbp/recruiter |
| 本月入职 | 本月已确认入职的候选人数 | admin/leader/hrbp |

**KPI 卡片规则**：
- 最多 4 个卡片一行
- 每个卡片包含：数字（大号）+ 标签（小号）+ 趋势箭头（↑↓→）
- 趋势箭头颜色：↑ 绿色（好转）、↓ 红色（恶化）、→ 灰色（持平）
- 点击 KPI 卡片跳转到对应列表页

### 2.3 页面分区

```
┌─────────────────────────────────────────────────────┐
│  👋 下午好，[角色名称]                                 │
│  今日概览：3 个待处理 · 2 场面试 · 1 个逾期反馈          │
├──────────┬──────────┬──────────┬────────────────────┤
│ 待处理    │ 在招岗位   │ 本周面试  │ 面试反馈逾期         │
│   3      │   12     │   8     │   2               │
│ ↑ 比上周+1│ → 持平    │ ↓ 比上周-3│ ↑ 比上周+1          │
├──────────┴──────────┴──────────┴────────────────────┤
│                                                     │
│  📋 我的待办 (My Actions)          📊 招聘动态         │
│  ┌─────────────────────┐  ┌──────────────────────┐  │
│  │ ⚠ 逾期: 补充面试反馈  │  │ 张三 提交了面试反馈    │  │
│  │    岗位: KA大客户销售  │  │ 李四 创建了画像校准    │  │
│  │    候选人: 林可       │  │ 王五 确认了校准结果    │  │
│  │    截止: 昨天         │  │ ...                 │  │
│  │ ⚠ 画像校准待确认      │  │                     │  │
│  │    岗位: 品牌策划     │  │                     │  │
│  │ ...                 │  │                     │  │
│  └─────────────────────┘  └──────────────────────┘  │
│                                                     │
│  🔔 风险提醒 (仅 admin/leader/hrbp)                    │
│  ┌──────────────────────────────────────────────┐   │
│  │ 🔴 KA大客户销售: 连续3人因行业经验不足淘汰       │   │
│  │ 🟡 品牌策划: 面试反馈质量偏低(平均分 62)         │   │
│  └──────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

### 2.4 信息分布规则

| 信息类型 | 位置 | 格式 |
|---------|------|------|
| 欢迎语 + 今日概览 | Page Header | 一句话，角色名来自 session |
| KPI 核心数字 | KPI Cards Row | MetricCard 组件 |
| 待办列表 | 左侧 60% 区域 | 紧凑列表，最多 5 条，溢出→查看全部 |
| 招聘动态 | 右侧 40% 区域 | ActivityTimeline，最新 5 条 |
| 风险提醒 | 底部横条 | ProfileSignalCard，按严重度排序 |

### 2.5 主操作

| 操作 | 按钮位置 | 行为 |
|------|---------|------|
| 查看全部待办 | 待办区域右上角 | 跳转 `/actions` |
| 查看全部动态 | 动态区域右上角 | 跳转动态列表 |
| 处理风险 | 风险卡片内 | 打开对应 Drawer/Modal |

## 3. Jobs 岗位管理

| **3 秒认知** | "哪些岗位在招？哪个岗位卡住了？" |
| **Phase** | Phase 3 |

### 3.2 核心 KPI

| KPI | 说明 |
|-----|------|
| 在招岗位 | 状态=ACTIVE 的岗位数 |
| 总候选人 | 所有岗位的候选人总数 |
| 平均招聘周期 | 从发布到 Offer 的平均天数 |
| 逾期岗位 | 超过目标到岗日期仍未关闭的岗位 |

### 3.3 列表设计

│ 岗位名称          │部门    │进度  │候选人│更新    │操作  │
│──────────────────│───────│─────│─────│───────│─────│
│ 🔴 KA大客户销售    │销售部   │🟡卡点│ 5/8 │2天前   │ ··· │

└─────────────────────────────────────────────────────┘

### 3.4 列表字段规则

| 列 | 对齐 | 数据层级 | 说明 |
|----|------|---------|------|
| 状态指示器 | 左 | 一级 | 红/绿/灰圆点 + 岗位名称 |
| 岗位名称 + 职级 | 左 | 一级 | 名称加粗，职级灰色小字在下 |
| 部门 + 负责人 | 左 | 二级 | 灰色正文 |
| 进度指示器 | 中 | 一级 | 进度条 + 候选人数(当前/目标) |
| 更新时间 | 右 | 三级 | 相对时间（"2天前"） |
| 操作 | 右 | 操作 | "···" 弹出菜单 |

### 3.5 列表 → Drawer 分工

| 信息类型 | 位置 |
|---------|------|
| 岗位基本信息（名称/部门/职级/负责人/JD摘要） | 列表行 + 悬浮 Tooltip |
| 完整 JD（职责+要求+加分） | Drawer · 概览 Tab |
| 岗位画像（mustHave/niceToHave/目标公司） | Drawer · 画像 Tab |
| 招聘漏斗（sourced→hired 各阶段人数） | Drawer · 漏斗 Tab |
| 业务反馈汇总（通过/不通过/原因分布） | Drawer · 反馈 Tab |
| 活动日志（谁在什么时候做了什么） | Drawer · 动态 Tab |

### 3.6 主操作

| 操作 | 触发方式 | 行为 |
|------|---------|------|
| 查看岗位详情 | 点击行 | 打开 JobDetailDrawer |
| 新建岗位 | 右上角按钮 | 打开 FormModal |
| 编辑岗位 | Drawer 内编辑按钮 | 切换为编辑模式 |
| 暂停/恢复岗位 | 行操作菜单 | 确认后更新状态 |
| 筛选岗位 | 分段控制器 + 搜索 | 前端过滤（数据量 < 100） |

## 9. Drawer / Modal / Tabs 使用规则

### 9.1 何时用 Drawer

| 场景 | Drawer 宽度 | 示例 |
|------|-----------|------|
| 详情查看（多信息） | `w-[640px]` | 候选人详情、岗位详情 |
| 详情查看（含 AI 分析） | `w-[720px]` | 候选人 AI 分析 |
| 简单详情 | `w-[480px]` | 面试反馈查看 |

**Drawer 规则**：
- 从右侧滑入，背景遮罩 `rgba(0,0,0,0.3)`
- 支持键盘 ESC 关闭
- 点击遮罩可关闭（除非有未保存内容）
- 内部使用 Tabs 管理多信息区块
- 最多 5 个 Tabs

### 9.2 何时用 Modal

| 场景 | Modal 宽度 | 示例 |
|------|-----------|------|
| 表单提交 | `max-w-lg` (512px) | 面试反馈、画像校准 |
| 确认操作 | `max-w-sm` (384px) | 确认删除、确认导入 |
| 简单创建 | `max-w-md` (448px) | 新建候选人 |

**Modal 规则**：
- 居中弹出，背景遮罩
- 支持键盘 ESC 关闭
- 有未保存内容时，关闭前弹出确认提示
- 一个 Modal 只做一件事
- 提交按钮必须有 loading 状态
- 错误信息在 Modal 内展示，不用 toast

### 9.3 何时用 Tabs

| 场景 | 位置 | Tab 数量 |
|------|------|---------|
| 岗位详情 | JobDetailDrawer | 5（概览/画像/漏斗/反馈/动态） |
| 候选人详情 | CandidateDetailDrawer | 4（概览/投递/面试/动态） |
| 面试详情 | InterviewDetailDrawer | 4（概览/反馈/质量/风险） |

**Tabs 规则**：
- Tab 切换不需要 loading（数据预加载）
- 默认打开第一个 Tab
- Tab 标签用名词，不用动词
- 不混合不同粒度的信息在同一 Tab

### 9.4 信息分布决策树

```
需要展示信息
├── 1-3 个字段 → 列表行内展示
├── 4-8 个字段 → 列表行 + Tooltip
├── 8-15 个字段 → Drawer（单 Tab）
├── 15+ 个字段 → Drawer（多 Tabs）
├── 需要用户输入 → Modal
├── 需要用户确认 → Modal（sm）
└── 需要分步操作 → Modal + StepIndicator
```

## 10. KPI 卡片业务规则

### 10.1 KPI 卡片结构

```
┌──────────────────────┐
│ 标签 (12px, 灰色)      │
│ 数字 (28px, 加粗, 深色) │
│ 趋势 (14px, ↑绿色/↓红色) │
└──────────────────────┘
```

### 10.2 趋势计算规则

| 趋势 | 条件 | 颜色 |
|------|------|------|
| ↑ 上涨 | 本周/本月 > 上周/上月 | `success` |
| ↓ 下降 | 本周/本月 < 上周/上月 | `danger` |
| → 持平 | 本周/本月 = 上周/上月 | `text-tertiary` |

趋势比较周期：
- 待处理/面试类：本周 vs 上周
- 招聘周期/效率类：本月 vs 上月

### 10.3 KPI 卡片点击行为

| KPI | 点击跳转 |
|-----|---------|
| 待处理事项 | `/actions` |
| 在招岗位 | `/jobs`（筛选 ACTIVE） |
| 总候选人 | `/candidates` |
| 待反馈 | `/interviews`（筛选 PENDING） |
| Offer 风险 | `/offer-risks` |

### 10.4 禁止的 KPI 做法

- ❌ 无动作的纯展示数字
- ❌ 无趋势对比的孤立数字
- ❌ 超过 4 个 KPI 卡片一行
- ❌ 装饰性图表（没有业务含义的饼图/柱状图）
- ❌ 假数据（如固定写死的 "98% 满意度"）

## 11. 禁止事项

### 11.1 全局禁止

| # | 禁止 | 原因 |
|----|------|------|
| 1 | **传统后台风格** | 不要只堆表格，不要政企蓝白灰 |
| 2 | **一个页面塞所有东西** | 用 Drawer/Modal/Tabs 分层 |
| 3 | **图表装饰化** | 每个图表必须有业务含义和交互 |
| 4 | **KPI 无动作** | 每个 KPI 卡片必须可点击跳转 |
| 5 | **假 AI 输出** | 不用 AI 就不要放 AI 区域 |
| 6 | **硬编码假人名** | 所有姓名从 DB 来，支持理然真实姓名 |
| 7 | **大屏驾驶舱风** | 不做军事化/作战指挥风格 |
| 8 | **raw JSON dump** | 错误不展示技术栈信息 |
| 9 | **undefined/null/NaN** | 所有字段有 fallback |
| 10 | **按钮一键多动作** | 一个按钮只做一个操作 |
| 11 | **暗色模式** | Phase 12 前不做 |
| 12 | **任意 hex 色值** | 只用 Design Token 定义的颜色 |

### 11.2 文案禁止

| 禁止词 | 推荐替换 |
|--------|---------|
| 作战/战报/军情 | 招聘复盘/数据概览 |
| 火力/攻坚/突击 | 重点推进/优先处理 |
| 指挥中心 | 工作台 |
| AI 决策/自动淘汰 | AI 辅助建议/建议复核 |
| 智能定薪 | 薪资建议（仅供参考） |
| 一键生成+确认+推送 | 分别操作：生成 → 确认 → 推送 |

### 11.3 页面禁止

| 禁止 | 正确做法 |
|------|---------|
| 列表页 + 详情全部展开在一页 | 列表页 + Drawer |
| Modal 里塞多个步骤的表单 | 一个 Modal 一个表单，多步骤用 StepIndicator |
| Tabs 超过 5 个 | 合并或拆分为子页面 |
| 表格列超过 8 列 | 次要信息进 Drawer 或 Tooltip |
| 筛选条件超过 5 个常驻 | 高频 3 个常驻 + 其余折叠为"高级筛选" |

## 12. UI Quality Gate

每个 UI Phase 交付前必须通过以下检查：

### 12.1 页面检查

- [ ] 页面四态（Loading/Empty/Error/PermissionDenied）全部实现
- [ ] 角色名称来自 session，不是硬编码
- [ ] 候选人/面试官姓名来自 DB，不是硬编码
- [ ] 所有数字有 fallback（不会显示 NaN）
- [ ] 所有日期使用相对时间格式（"2天前"，不是 raw timestamp）
- [ ] 页面标题 + 描述清晰

### 12.2 组件检查

- [ ] 没有使用任意 hex 色值
- [ ] 交互元素有 hover/active/focus 态
- [ ] 按钮有 loading 状态
- [ ] 表单有 validation error 展示
- [ ] 可关闭的 Drawer/Modal 支持 ESC 键

### 12.3 数据检查

- [ ] 列表数据按 scope 过滤（不同角色看到不同数据）
- [ ] API 响应有 error handling
- [ ] 空数据展示 EmptyState，不是空白页
- [ ] 权限不足展示 PermissionDenied，不是 500 错误
