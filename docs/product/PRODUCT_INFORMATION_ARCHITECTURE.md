## 1. 产品概述与核心价值主线

### 1.1 产品定位

**理然智能招聘 AI 看板** 是一个轻量级 SaaS 招聘工作台。它不是 Moka 替代品，也不是传统招聘后台。

它要解决的核心问题是招聘中的 **判断与协同问题**：

| # | 问题 | 当前痛点 |
|---|------|---------|
| 1 | 岗位到底为什么卡住 | 看数据不知道根因，只知道"招不到人" |
| 2 | 候选人为什么推进或淘汰 | 淘汰原因散落在微信/邮件/口述中 |
| 3 | 面试反馈是否有证据 | 反馈只有"感觉不错"，没有具体行为证据 |
| 4 | 面试官是否按岗位画像提问 | 面试官自由发挥，偏离岗位核心要求 |
| 5 | 业务反馈是否超时 | 业务负责人拖延反馈，HR 无法追踪 |
| 6 | Offer 风险是否提前预警 | Offer 阶段失联/拒签后才被动发现 |
| 7 | HR 的跟进动作是否闭环 | 发现风险 → 做了什么 → 结果如何，无留痕 |
| 8 | 招聘项目是否可复盘 | 每个岗位招完后没有结构化数据可回溯 |
| 9 | AI 能否帮助 HR 提前发现问题 | 数据在手但靠人肉逐条筛查 |

### 1.2 核心价值主线

所有模块必须服务于以下主线：

```
招聘数据沉淀
    ↓
AI / 规则识别风险
    ↓
形成结构化判断
    ↓
生成行动项 (ActionItem)
    ↓
责任人处理 (Create/Resolve/Dismiss)
    ↓
Activity 留痕 (ActivityLog)
    ↓
管理层复盘 (ReportSnapshot)
```

如果某个模块不能服务这条链路，**不做**。

### 1.3 产品边界

**做什么**：
- 招聘效率分析与漏斗诊断
- 候选人全链路管理
- 面试质量评估与结构化反馈
- Offer 风险提醒与入职前承接
- 协同 Action 任务中心
- AI 辅助分析（识别盲点，不替代判断）
- 周期性复盘与趋势报告

**不做什么**：
- 不替代 Moka / ATS（不做简历库/网申/offer 审批流）
- 不发 Offer / 不做电子签
- 不做薪酬 / 合同 / 入职
- 不自动录用 / 淘汰 / 推进候选人
- 不做 AI 自动决策（AI 只建议，不执行）

## 3. 模块详细定义

### 3.1 招聘总览（AI 招聘洞察看板）(`/dashboard`)

| 属性 | 值 |
|------|-----|
| **模块定位** | 招聘全局概览，一站式发现风险与待办。用户登录后第一个页面 |
| **核心用户** | HRBP、招聘负责人、管理层 |
| **解决的招聘问题** | "我今天该处理什么？招聘整体怎么样？有没有需要我关注的风险？" |
| **3 秒认知** | 数字 + 趋势 + 待办列表 + 风险提醒 |

**核心 KPI**：

| KPI | 说明 | 数据来源 |
|-----|------|---------|
| 待处理 Action 数 | 当前用户需要处理的 Action 数量 | ActionItem (status=pending) |
| 在招岗位数 | 状态=open 的岗位数 | Job (status=open) |
| 本周面试数 | 本周安排的面试数 | Interview (scheduledAt in this week) |
| 逾期反馈数 | 已面试但超 24h 未提交反馈数 | InterviewFeedback (submittedAt=null) |
| Offer 风险数 | 当前有风险的 Offer 数 | OfferRisk (status=open) |

**关联对象**：ActionItem, Job, Candidate, Interview, OfferRisk, ActivityLog

**Action 能力**：是（看板展示从规则引擎自动生成的风险 Action，也支持手动创建）

**AI 能力**：
- Layer 1：规则引擎自动识别逾期、卡点、风险信号
- Layer 2：AI 辅助趋势分析（"本月面试通过率下降 12%"）

**当前状态**：空状态（模块已定义，Phase 9+ 实现）
**后续优先级**：P0

### 3.2 招聘风险行动中心 (`/actions`)

| 属性 | 值 |
|------|-----|
| **模块定位** | 所有招聘风险的统一处理入口。从发现问题到闭环处理的全链路 |
| **核心用户** | HRBP、招聘专员 |
| **解决的招聘问题** | "有哪些事需要我处理？怎么闭环？谁在处理？进度如何？" |
| **3 秒认知** | 待处理数字 + 逾期标记 + 分类筛选 + 一键处理 |

**核心 KPI**：

| KPI | 说明 | 数据来源 |
|-----|------|---------|
| 待处理 Action | status=pending 的 Action 数 | ActionItem |
| 逾期 Action | pending 且 dueAt < now 的数量 | ActionItem |
| 今日到期 | pending 且 dueAt = today | ActionItem |
| 本周解决数 | 本周内 resolvedAt 不为空的 Action 数 | ActionItem |

**关联对象**：ActionItem, Job, Candidate, Application, Interview, ActivityLog, User

**Action 能力**：是（本身即 Action 管理模块：Create / Resolve / Dismiss + ActivityLog 留痕）

**AI 能力**：
- Layer 1：规则引擎自动生成 Action（逾期、卡点、风险信号）
- Layer 2：AI 辅助生成 Action 描述和建议处理方案（Phase 10+）

**当前状态**：🟢 已交付（Phase 7.D Final Lock + Patch）
**后续优先级**：P0（持续迭代）

### 3.3 岗位中心 (`/jobs`)

| 属性 | 值 |
|------|-----|
| **模块定位** | 岗位全生命周期管理 + 卡点诊断。从岗位创建到关闭的全流程 |
| **核心用户** | HRBP、招聘负责人 |
| **解决的招聘问题** | "哪些岗位在招？哪个卡住了？为什么卡住？需要调整什么？" |
| **3 秒认知** | 岗位列表 + 进度指示器 + 卡点标记 + 候选人漏斗 |

**核心 KPI**：

| KPI | 说明 | 数据来源 |
|-----|------|---------|
| 在招岗位数 | status=open 的岗位数 | Job |
| 平均招聘周期 | 从 open 到 closed 的平均天数 | Job (createdAt → closedAt) |
| 逾期岗位数 | 超过目标到岗日期仍未关闭的岗位 | Job (dueDate < now) |
| 候选人漏斗转化率 | sourced→hired 各阶段转化率 | Application (stage transitions) |

**关联对象**：Job, Application, Candidate, Interview, BusinessFeedback, ProfileCalibration, ActionItem

**Action 能力**：是（岗位卡点自动生成 Action：候选人不足、画像需校准、业务反馈逾期等）

**当前状态**：空状态
**后续优先级**：P1

### 4.1 Action 产生与消费关系

```

                    │  招聘总览  │ ← 消费所有模块的 Action + KPI
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
```

## 7. 设计约束与红线

### 7.1 架构红线

- ❌ 不允许模块间循环依赖
- ❌ 不允许模块绕过 ActionItem 直接调用其他模块的 mutation
- ❌ 不允许模块自行定义数据格式（统一使用 Prisma Schema 定义的对象）
- ❌ 不允许模块在页面内直接操作 DOM（使用统一的 UI 组件基座）
- ❌ 不允许模块使用独立的设计 token（统一使用 Design System）

### 7.2 数据红线

- ❌ 不允许硬编码业务数据（所有人名/岗位名/数据来自 DB）
- ❌ 不允许前端硬编码 API URL（统一使用 API client）
- ❌ 不允许直接暴露 Prisma model 给前端（通过 API route 和 scope 过滤）
- ❌ 不允许跨角色数据泄露（严格按 scope 过滤）

### 7.3 AI 红线

- ❌ 不允许 AI 自动执行操作（AI 只建议，用户必须手动确认）
- ❌ 不允许 AI 输出不带来源和置信度
- ❌ 不允许 AI 替代面试官判断
- ❌ 不允许 AI 在无数据支撑时"编造"分析结果
